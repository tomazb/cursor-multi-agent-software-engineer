# Design: v0.2 Local Hardening and Git Isolation

**Issue:** GitHub #2 (ROADMAP milestone v0.2)  
**Date:** 2026-07-22  
**Status:** Approved for implementation (user requested brainstorm + implement)

## Problem

v0.1 delivers a working local orchestrator but lacks durability, isolation, and fail-closed hardening required before a GitHub App pilot: the file store can tear under concurrent writes, artifact digests are not revalidated, same-name artifacts overwrite history, git provenance is not persisted, builder edits happen in the ambient checkout, secrets can leak into artifacts, prompts travel on argv, failed runs are terminal, and configuration lacks machine-checkable schemas.

## Goals

Deliver ROADMAP v0.2 end-to-end so local runs are auditable, recoverable, and safer under concurrent access and untrusted model output—without introducing hosted control-plane or GitHub App scope (those are v0.3+).

## Non-goals

- GitHub webhooks, check runs, authenticated PR automation
- PostgreSQL / object storage
- Automatic merging or multi-tenant policy hierarchy
- Provider sandboxing beyond existing read-only fingerprint checks

## Approaches considered

### A — Split into many micro-PRs (one subsystem each)

**Pros:** Small reviewable diffs.  
**Cons:** Leaves the milestone incomplete; hard to keep contracts consistent across half-landed store/git changes.  
**Rejected** for completing issue #2 in one delivery.

### B — Single cohesive v0.2 package with clear module boundaries (recommended)

**Pros:** Matches the roadmap milestone; keeps store, git, markers, redaction, and recovery consistent; one version bump story.  
**Cons:** Larger PR; requires careful task sequencing and tests.  
**Chosen.**

### C — Store + markers only; defer worktrees/commits/schemas

**Pros:** Smaller.  
**Cons:** Leaves documented architecture gaps and ADR-0002/0005 follow-ups open; does not close issue #2.  
**Rejected.**

## Design

### 1. Persistence: `RunStore` interface + atomic `FileRunStore`

Extract `RunStore` from the concrete class. Persist each save via write-to-temp + `rename`, with:

- `version: number` on `RunRecord` (optimistic concurrency; save fails if on-disk version ≠ expected)
- Advisory lock file `.maswe/runs/<id>/.lock` held during mutate operations
- Artifact writes also atomic; digests recomputed and compared on every `readArtifact`
- Attempt-scoped immutable artifact names: `<logicalName>.attempt-<n>.md` plus a stable logical pointer in `run.artifacts` (`logicalName` + `attempt`)

Consumers (`Orchestrator`, CLI, prompt-builder) depend on the interface.

### 2. Git provenance on the run

Extend `RunRecord` with:

```ts
workspace: {
  baseSha: string;
  headSha: string;
  branch: string;
  fingerprint: string;
  worktreePath?: string;
}
```

Captured at run start from the repository (or created worktree). Updated after deterministic commits. Fingerprint continues to gate read-only roles.

### 3. Worktree / branch manager + deterministic commits

New module `src/git-workspace.ts` owned by deterministic code (ADR-0005):

- `ensureRunWorkspace(run)` — create `maswe/<runId>` branch and linked worktree under `.maswe/worktrees/<runId>` when `policy.useIsolatedWorktree` is true (default **true** for new configs; mock tests may disable)
- `createDeterministicCommit(cwd, message, { allowedPaths? })` — stage only tracked/intended paths, reject paths outside change-scope globs from design artifact or config `policy.allowedPathGlobs` (default `**` for v0.2 local use, with explicit deny for `.maswe/config.json` secrets patterns)
- Scope check compares `git diff --name-only baseSha..HEAD` against allowed globs

Orchestrator builder/resolver stages execute against `run.workspace.worktreePath ?? repositoryPath`.

### 4. Verifier defects → builder

On `VERIFY_FAILED`, parse blocking findings from the verification report into `10-verifier-defects.attempt-<n>.md` (structured Markdown list). Builder prompt gains `{{VERIFIER_DEFECTS}}` (latest attempt). Keep full report available as today.

### 5. Strict terminal markers

Shared `src/markers.ts` validates required end markers before accepting a role success event:

| Role | Required marker |
|------|-----------------|
| brainstormer | `READY_FOR_BRAINSTORM_APPROVAL` |
| designer | `READY_FOR_DESIGN_APPROVAL` |
| builder | `BUILD_COMPLETE` |
| verifier | `VERDICT: PASS` or `VERDICT: FAIL` |
| classifier | `SCOPE: IN_SCOPE` or `SCOPE: OUT_OF_SCOPE` |
| prResolver | `RESOLUTION_COMPLETE` |

Missing marker → fail the role attempt (retry/fallback models still apply).

### 6. Secret redaction

`src/redaction.ts` strips common secret patterns (API keys, tokens, PEM blocks, `Authorization:` headers, AWS keys) before artifact persistence and before embedding process stderr into quality reports. Redaction is best-effort and never throws away the artifact—only masks matches.

### 7. Stdin prompt transport

Cursor CLI runtime prefers `--print` with prompt on stdin when `policy.promptTransport` is `"stdin"` (default) or when argv would exceed a safe length; falls back to argv with a doctor warning when stdin unsupported. Implementation: pass prompt via `spawnCaptured` `input`, omit trailing argv prompt, use `-p` still for print mode.

### 8. Budgets and timeouts

Extend `policy`:

- `commandTimeoutMs` (default 600_000)
- `roleTimeoutMs` (default 1_800_000)
- `maxRunDurationMs` (optional; default unset)

`spawnCaptured` kills the child on timeout. Orchestrator fails the run if wall-clock since `createdAt` exceeds `maxRunDurationMs`.

### 9. Retry-from-failed and supersede

New events (centralized in `state-machine.ts`):

- `RETRY_FROM_FAILED`: `FAILED` → resume state stored in `failure.resumeState` (captured on FAIL)
- `SUPERSEDE`: any non-terminal (or FAILED/CANCELLED) run may be marked `supersededBy`; creates a new run copying title/request/config and linking `supersedes`

CLI: `maswe retry <run-id>`, `maswe supersede <run-id>`.

### 10. JSON schemas

Add `schemas/config.schema.json` and `schemas/run-record.schema.json`. Config load optionally validates when `ajv`-free: use a minimal hand-rolled schema check aligned with TypeScript types (no new runtime dependency)—or embed JSON Schema files for documentation + a lightweight validator that checks required keys/types. Prefer **zero new dependencies**: ship schemas as docs/CI artifacts and keep `assertConfig` as the runtime enforcer; add a test that DEFAULT_CONFIG and sample artifacts satisfy the schema via a small internal checker.

### 11. Packaged CLI release and lock file

- Keep `package-lock.json` committed; CI uses `npm ci`
- Add `npm pack` dry-run to CI or a `scripts/verify-pack.mjs` invoked from `check` optionally
- Document release steps; bump version to `0.2.0` in package.json and CHANGELOG

## Testing

- Unit: store atomicity/version conflict/digest mismatch; markers; redaction; git scope; timeouts
- Workflow: verifier fail → defects artifact → builder retry; retry-from-failed; supersede; isolated worktree path used when enabled
- Config: new policy fields merge + validation
- Preserve existing orchestrator happy-path tests (adjust mock markers if needed)

## Risks

- Worktree creation requires a git repo; mock tests must init git or set `useIsolatedWorktree: false`
- Cursor CLI stdin support may vary; keep argv fallback
- Optimistic concurrency may surface in multi-process CLI use—fail closed with clear error
